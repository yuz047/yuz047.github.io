GreenLight Trader: a $1,000 SPY-anchored alpha overlay.

Mandate: beat SPY by 10%, trail SPY by no more than 5%. The book holds SPY as the always-on baseline. A multi-factor pitcher identifies high-conviction names (composite z ≥ 1.0) and substitutes them for portions of the SPY core. When a pick underperforms SPY by more than 2% on a closing basis, it's force-exited and the SPY allocation is restored. Strategy modules are hot-swappable; the EOD reviewer can propose changes but cannot apply them. Not investment advice.

What it does

From OHLCV to a defensible next-day shortlist.

Free OHLCV from yfinance plus Yahoo Finance RSS headlines feed a daily pipeline. The pipeline computes signals once, runs every active strategy across every symbol, ranks the resulting candidates by composite score, and pushes them through a risk gate. The risk gate is the final word: it rejects a candidate if it would breach the 1% per-trade or 10% portfolio drawdown limits, if the name is already in the book, or if the system is in red or black state. Whatever survives is opened on tomorrow's open in the backtest and on today's close in the live job. Every trade keeps its rationale; the EOD review can call an LLM (Claude Haiku by default) to write the daily wrap or, with no API key set, fall through to a deterministic template that fills in the same fields.

Architecture

Engine writes, dashboard reads.

One source of truth — the Python engine — and a public read-only dashboard. The cron runs daily on a GitHub Actions runner; it pulls fresh data, recomputes signals, advances the book, writes JSON snapshots to the repo, persists the paper-book state, and (optionally) mirrors them into a Supabase Postgres database. The dashboard is a static page that fetches those snapshots over HTTPS.

Hot-swappable logic

Strategies are files, not branches.

Every strategy lives in its own module under python/strategies/. Each file exposes a manifest (id, version, rules, parameters) and a run() entrypoint. On startup the package auto-discovers every module — no central register to edit. To activate a new strategy, list its id in config.ACTIVE_IDS. To upgrade, drop a file with a bumped version next to the old one and switch the id. Old versions stay readable for audit and back-comparison; nothing mutates rules at runtime.

Why the friction is the feature

Friction stops the system from chasing today's losing trade with tomorrow's new rule. The EOD reviewer can propose a change (logged in the decision log below), but it cannot apply one. A human runs a comparison backtest, accepts or rejects, and bumps the version. This is how strategy drift is controlled.

Risk framework

Relative to SPY, not absolute.

The V2 risk gate is benchmarked. The mandate is to beat SPY by 10% over two years and to never trail SPY by more than 5%. The system enforces this at three levels: each pick has a 2% relative stop vs SPY (kills the pick if it lags); the portfolio has a 4% relative-DD yellow gate (reduces to one pick) and a 4% red gate (closes all picks, holds 100% SPY for 20 trading days); per-pick weight is capped at 20% of NAV, floor at 60% SPY when any picks are active.
